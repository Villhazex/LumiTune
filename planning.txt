# Level 1 + Level 2 Metadata Split — Implementation Plan

## Architecture

Separate metadata into TWO layers:

| Layer | Speed | Source | When |
|---|---|---|---|
| **Level 1** (Display) | Instant | ID3 + filename + folder | During scan → show immediately |
| **Level 2** (Canonical) | Background | fpcalc + AcoustID + MusicBrainz | Worker queue → enrich progressively |

Display metadata lives in `files` table columns.
Canonical metadata stays in `metadata_cache` + `identifications` (as-is).

### A6 — Background Enrichment (new or merged into runQueueCollect)

```js
async function startBackgroundEnrichment(files, plKey) {
  await inv('start_queue', { acoustidKey, concurrency: 3 });
  while (true) {
    await sleep(250);
    const qs = await inv('get_queue_status');
    if (!qs.running) break;

    const drain = await inv('drain_processed');
    for (const r of drain) {
      if (!r.success) continue;
      // Find song by filePath and update its title/artist/album
      const song = Object.values(songs).find(s => s.filePath === r.path);
      if (!song) continue;
      if (song.metadataSource === 'manual') continue; // NEVER overwrite manual edits
      applyCanonicalUpdate(song, r);
    }
    updateUI(); // refresh visible rows
  }
  await inv('stop_queue').catch(() => {});
}

function applyCanonicalUpdate(song, result) {
  // Source priority: if current is ID3 and canonical is AcoustID with ≥0.9, update
  // If current is filename and canonical is anything ≥0.4, update
  const priorities = { manual: 5, acoustid: 4, id3: 3, hybrid: 2, filename: 1, folder: 0 };
  const currentPrio = priorities[song.metadataSource] || 0;
  const newPrio = priorities[result.method] || 0;

  if (newPrio > currentPrio || (newPrio === currentPrio && result.confidence >= 0.9)) {
    const oldTitle = song.title;
    song.title = result.title || song.title;
    song.artist = result.artist || song.artist;
    song.album = result.album || song.album;
    song.genre = result.genre || song.genre;
    song.year = result.year || song.year;
    if (result.cover_data_base64) {
      song.cover = 'data:' + result.cover_mime + ';base64,' + result.cover_data_base64;
    }
    song.metadataSource = result.method;
    song.reliability = result.reliability;
  }
}
```

## Session B: Metadata Merge Strategy (Rust + JS)

### B1 — Merge Rules (new module or add to metadata.rs)

Principles:
1. **Manual edits NEVER overwritten** (new `metadata_source` field)
2. **Safe fields**: genre, year, cover art, MusicBrainz IDs — auto-update on any canonical result
3. **Protected fields**: title, artist, album — update only if new source has higher priority
4. **Priority ladder**: manual(5) > acoustid(4) > cache/audio_hash(3.5) > hybrid(3) > id3(3) > filename(2) > folder(1) > none(0)

Priority system in Rust:

```rust
fn source_priority(source: &str) -> u8 {
    match source {
        "manual" => 5,
        "acoustid" => 4,
        "cache" | "audio_hash" => 3,
        "id3" | "hybrid" => 3,
        "filename" => 2,
        "folder" => 1,
        _ => 0,
    }
}

fn should_update_display(current_source: &str, new_source: &str, new_confidence: f64) -> bool {
    let cur = source_priority(current_source);
    let new = source_priority(new_source);
    new > cur || (new == cur && new_confidence >= 0.9)
}
```

### B2 — JS Merge Logic (settings.js or new helpers.js functions)

`applyCanonicalUpdate()` as defined above, with the priority + confidence check.

## Session E: metadata.rs Module Refactor

Split monolithic `metadata.rs` (868 lines) into modules:

```
src-tauri/src/
  metadata/
    mod.rs          — re-exports + public API (identify_file)
    display.rs      — read_id3, has_minimal_tags, is_garbage_tags, Id3Tags
    fpcalc.rs       — fpcalc_bin, run_fpcalc
    acoustid.rs     — lookup_acoustid, AcoustidResult/Hit/Recording/Artist
    musicbrainz.rs  — lookup_musicbrainz, fetch_cover, MbRecording/Release/etc
    normalize.rs    — normalize_title, titles_match, parse_filename
    merge.rs        — metadata merge strategy
    queue.rs        — IdentificationResult, IdentificationProgress, reliability, save_cover
```

File sizes after split:
- `normalize.rs`: ~60 lines (normalize_title, titles_match, parse_filename)
- `display.rs`: ~100 lines (read_id3, Id3Tags, has_minimal_tags, is_garbage_tags)
- `fpcalc.rs`: ~60 lines (fpcalc_bin, run_fpcalc)
- `acoustid.rs`: ~60 lines (lookup_acoustid + structs)
- `musicbrainz.rs`: ~100 lines (lookup_musicbrainz, fetch_cover, structs)
- `merge.rs`: ~50 lines
- `queue.rs`: ~50 lines
- `mod.rs`: ~40 lines (identify_file, re-exports)

> **Note**: Do NOT move `throttle_api()` into its own tiny file — it's used by both musicbrainz and cover art. Keep it accessible (e.g., in `mod.rs` or a small `rate_limit.rs`).

## Testing Strategy

1. **Scan folder with known mp3s** → verify display_title/display_artist populated instantly
2. **Create playlist** → verify songs appear with Level 1 data (no wait)
3. **Background enrichment** → verify title/artist progressively improve
4. **Manual edit** → verify metadata_source='manual' and enrichment doesn't overwrite
5. **Module split** → verify `cargo build` succeeds and `cargo t` passes
6. **Edge: no ID3 tags** → verify filename parsing works
7. **Edge: swapped tags** → verify auto-detect + fix works
8. **Edge: empty library** → verify no crash
